ClinicApp — Clinic Appointment Management System
ClinicApp is a RESTful Web API built with ASP.NET Core 8 following Clean Architecture principles, designed to manage doctors, patients, and appointments in a clinical environment. The solution is divided into four distinct layers: Domain, Application, Infrastructure, and API, each with a clear and single responsibility. The Domain layer sits at the core and has zero external dependencies, containing only entities, enums, and repository interfaces. The Application layer depends solely on Domain and holds all business logic through service classes, DTOs, and service interfaces. The Infrastructure layer implements the repository interfaces defined in Domain using Entity Framework Core with SQL Server as the database provider. The API layer is the outermost layer, containing ASP.NET Core controllers that delegate all work to Application services via dependency injection.
The system manages three primary entities: Doctor, Patient, and Appointment. A Doctor has properties such as first name, last name, specialization, phone, email, and availability status. A Patient stores personal information including date of birth, contact details, and address. An Appointment links a Doctor and a Patient together with a scheduled date, optional notes, and a status that can be Pending, Confirmed, Cancelled, or Completed. All entities inherit from a shared BaseEntity class that provides Id, CreatedAt, and UpdatedAt fields automatically. The Appointment entity uses a custom enum called AppointmentStatus to represent its current state in a type-safe manner.
The repository pattern is applied generically through a base IRepository interface that exposes GetByIdAsync, GetAllAsync, AddAsync, Update, Delete, and SaveChangesAsync. Each entity then has its own extended repository interface — IDoctorRepository, IPatientRepository, and IAppointmentRepository — that adds domain-specific query methods. For example, IDoctorRepository exposes GetBySpecializationAsync and GetAvailableDoctorsAsync, while IAppointmentRepository exposes GetByDoctorIdAsync, GetByPatientIdAsync, GetByDateAsync, and GetWithDetailsAsync which eager-loads related Doctor and Patient data using EF Core Include. The concrete implementations of these interfaces live in the Infrastructure layer and extend a GenericRepository base class, keeping common CRUD logic in one place and avoiding duplication.
Each entity has three corresponding DTO classes — a read DTO, a create DTO, and an update DTO — keeping API input and output contracts clean and decoupled from domain models. The AppointmentDto, for instance, flattens the related Doctor and Patient names directly into the response so the client does not need to make additional requests. Service classes in the Application layer handle all mapping between entities and DTOs manually without any third-party mapping library, making the data flow completely explicit and easy to trace. Dependency injection is configured through two static extension methods, AddApplication and AddInfrastructure, which are called in Program.cs to keep the startup file minimal and readable.
The API exposes fully RESTful endpoints across three controllers: DoctorsController, PatientsController, and AppointmentsController. Every controller follows the same consistent pattern — returning 200 OK for successful reads, 201 Created for successful creates with a location header, 204 NoContent for successful deletes and cancellations, and 404 NotFound when a resource does not exist. The AppointmentsController includes a dedicated PATCH endpoint at /api/appointments/{id}/cancel that sets the appointment status to Cancelled without requiring a full update payload. Swagger UI is enabled in development mode and is accessible at /swagger, providing interactive documentation for all endpoints out of the box. The database connection string is stored in appsettings.json and targets SQL Server LocalDB by default, making it ready to run locally without any additional database setup beyond running EF Core migrations.
